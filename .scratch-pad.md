## TikTok Puzzle CAPTCHA Solver — Key Learnings (2026-03-13)

### Popup Detection
- Dimmed overlay: brightness ~128 at screen edges (x=50) when CAPTCHA popup present
- Popup: white (255) center, roughly x=120-1060 pixels (WDA 40-353)
- Photo area within popup: center brightness < 230
- Slider track: white area below photo, slider_y ≈ WDA 520 (varies slightly)
- Slider handle starts at left: WDA x ≈ 67

### Solving Strategy
- **Multi-attempt slider at fixed percentages works best**: try [0.45, 0.50, 0.55, 0.60, 0.40, 0.65, 0.35, 0.70]
- Edge detection for piece/cutout is unreliable (photo content confuses it)
- Each drag attempt: 5s timeout + 3s wait = ~8s per attempt
- CAPTCHA timeout: 2-3 minutes, enough for 5-8 attempts
- TikTok presents 3-5 consecutive CAPTCHAs per signup
- Overshooting works — piece snaps into cutout

### WDA Timing Issues
- `dragfromtoforduration` response takes ~57s (TikTok quiescence)
- Use 5s timeout on separate httpx.Client — gesture still executes
- `find_element` also times out with TikTok open
- Screenshots work with 30-45s timeout
- W3C actions (tap) also slow: use 10s timeout, gesture executes

### Email Entry
- MUST use `wda/keys` (character-by-character) instead of `element_value`
- Autocomplete bar (gmail/yahoo/outlook) corrupts text if element_value used
- Clear field: go back to birthday → forward again (element_clear times out)
- Alternative: backspace characters via wda/keys (50x `\u0008`)

### Continue Button Detection
- Use `_find_wide_red_band()` — returns WDA y coordinate of button center
- Birthday Continue: y ≈ 727 (NOT 770 as initially assumed)
- Email Continue: y ≈ 470
- Loading state: red button with spinner, still detects as red band

### Current Status
- testaccount2026@outlook.com was used for testing, likely rate-limited
- CAPSOLVER_API_KEY and TWOCAPTCHA_API_KEY are empty in .env

## mail.tm REST API — Email Provider (2026-03-13)

### Key Facts
- REST API at `https://api.mail.tm` — no browser/CAPTCHA needed
- Active domain: `dollicons.com` (changes periodically)
- Accounts created via `POST /accounts`, auth via `POST /token` (JWT)
- Messages read via `GET /messages` with Bearer token

### Critical Bug: Dot Normalization
- **mail.tm strips dots from the local part of email addresses**
- e.g., `emily.carter.94.d5db13ucp@dollicons.com` becomes `emilycarter94d5db13ucp@dollicons.com`
- The `/token` endpoint does EXACT matching on the normalized address
- **FIX**: Always use the `address` field from the 201 response, not what you sent
- Accounts created before this fix have unusable tokens (password mismatch on lookup)

### Rate Limits
- ~2-3 accounts per 10 seconds before hitting 429
- 8s delay between accounts + exponential backoff on 429 works reliably
- 50 accounts takes ~7 minutes

### Integration Notes
- Stored in `email_accounts` with provider="mailtm", imap_host="api.mail.tm", imap_port=0
- No IMAP — must use REST API for reading verification codes
- `email_api.py:poll_for_code_mailtm()` replaces IMAP-based `poll_for_code()`
- Account creator (`persona/account_creator.py`) uses `ImapConfig` — needs update to support mailtm provider

### mail.com Signup — DIRECT API APPROACH (solved)
- Angular 21 button/form approach abandoned — zoneless CD blocks CaptchaFox callback propagation
- **Working approach**: Navigate signup steps → extract service config → solve captcha → call registration API directly
- Registration API: `POST {accountRestUrl}/email-registration` with exact headers:
  - `Content-Type: application/vnd.ui.mam.account.creation+json`
  - `X-UI-APP: {appIdentifier}`, `X-CCGUID: {clientCredentialGuid}`
  - `X-REQUEST-ID: {uuid}`, `Template-Name: {templateName}`
  - `Authorization: {authToken}` (Bearer token from Angular config)
  - `cf-captcha-response: {captcha_token}` (CaptchaFox-specific)
- Service config extracted from Angular injector: `getAllAngularTestabilities()[0]._destroyRef.records` → find service with `createAccount` method
- Store data (email, password, name, DOB, etc.) from NgRx store: find `dispatch` method in records → traverse `source._value` chain
- Response: 204 = success, 403 = rate limited
- Post-registration: submit hidden `onereg-login-form` → click `#continueButton` → inbox
- **IMAP/POP3 is Premium-only** — email reading MUST use web interface (3c-lxa.mail.com iframe)
- Rate limiting: ~6 accounts before 403 from same IP. Need proxy rotation for bulk creation.
- Studio IP currently rate-limited (as of 2026-03-13)
- Hetzner VPS (5.78.157.238): 502 — mail.com blocks datacenter IPs with "mam.registration:server-error"
- Tor exit nodes: blocked ("country where registration is no longer possible")
- Need **residential proxy** to bypass both rate limit AND datacenter IP blocking
- Hybrid approach ready: browser runs on studio (fast captcha), API call via proxy/ssh_host

### mail.com API Body Format Fix (2026-03-13)
- **CRITICAL**: The API expects nested body from `helper.getUserInfoFrom()`, NOT flat store fields
- **Wrong** (what we were sending): `{firstName, lastName, password, country, salutation, dateOfBirth}`
- **Correct** (what API expects): `{givenName, familyName, gender, birthDate, credentials: {password}, address: {countryCode, region}}`
- Fixed in `email_playwright.py:_extract_store_data()` — now uses Angular's `helper.getUserInfoFrom()` method
- The helper is found via effect service with `submitFreemail` key
- `createCaptchaData(token, userSolution)` also from helper — maps `{token: getValueFrom(captcha.token), userSolution: getValueFrom(captcha.userSolution)}`

### Phone WebInspector Pipeline (2026-03-13)
- iPhone-B (UDID: 00008140-001975DC3678801C, WDA port 8100)
- Web Inspector enabled via WDA tap automation on Settings → Safari → Advanced
- `pymobiledevice3.services.webinspector.WebinspectorService` for JS execution
- Async API: must run in separate thread (ThreadPoolExecutor) to avoid Playwright event loop conflict
- Page discovery: `get_open_pages()` returns dict `{app_name: [Page objects]}`
- App open: `inspector.open_app("com.apple.mobilesafari")` (bundle ID, not display name)
- Session: `inspector.inspector_session(app, page)`
- JS execution: `session.runtime_evaluate(js_string)` — result varies (str or dict)
- Polling pattern: `window.__variable` for async results
- Phone must have Safari on `signup.mail.com` for same-origin fetch to work
